课堂讲义：http://t.cn/RQs9iTw

微服务定义

Martin Fowler的博客中对微服务架构的定义要点：

  1. 一种架构风格，将单体应用划分成一组小的服务，服务之间相互协作，实现业务功能；
  2. 每个服务运行在独立的进程中
  3. 服务间（进程间）采用轻量级的通信机制（HTTP或RPC）
  4. 基于业务能力构建或划分（用户服务、商品服务、店铺服务…）
  5. 各服务由负责的团队进行独立的、自动化的部署，团队之间耦合度不高
  6. 无集中式的服务管理，每个团队可以采用自己擅长的技术栈，可以使用不同的开发语言、不同的存储机制

参考博客地址：https://www.martinfowler.com/articles/microservices.html

前Netflix架构总监，经过Netflix由单体应用转为微服务的整个演化的过程，提出了如下对微服务的抽象定义: 基于有界上下⽂的，松散耦合的⾯向服务的架构

  1. Loosely Coupled 服务可以独立的开发部署，不强依赖于其他服务
  2. Service Oriented Architecture 本质上还是SOA，是更细粒度的SOA
  3. with bounded context 有界上下文、或局部状态，每个团队或每个服务有自己的独立数据源

架构师如何权衡微服务的利弊？

微服务的优势：

  1. 强模块化边界 模块化演进：类、包 –> 类库/组件 –> 服务
  2. 可独立部署
  3. 技术多样性

微服务带来的问题：

  1. 分布式的复杂性：服务非常多，成百上千个服务
  2. 最终一致性：每个团队可能有自己的数据拷贝
  3. 运维复杂性：服务监控、容量规划、可靠性、稳定性，对运维的挑战极大
  4. 测试复杂性：服务之间有协作时，需要协作测试

康威法则

设计系统的组织，其产生的设计等同于组织之内、组织之间的沟通结构。用更容易理解的话来说，就是：组织形式等同系统设计。

公式刚起步时，处于探索阶段，探索其模式是否有效，此时往往只有很小的开发团队，应用也是单体应用。但是随着公司业务的进步与发展，开发人员越来越多，逐渐划分为不同的团队。如果此时还是单体应用的话，就违反了康威法则，因为单体应用的设计没有体现出组织架构。

这时候的解决方案是什么呢？微服务。通过微服务化，将单体应用拆分为多个应用，每个团队负责自己的应用，遵循了康威法则。

所以，架构师不仅要做技术架构，同时也要掌握组织架构。

企业应该在什么时候开始考虑引入微服务

微服务的适用性：企业应该在什么时候开始考虑引入微服务？

个人经验：公式处于初创阶段时，单体应用架构优先。当团队达到百人级别时，就可以考虑引入微服务架构。

架构：考虑单块应用优先。微服务需要前期的基础设施搭建，成本较大，而公司也处于初步探索阶段，商业模式有待验证，此时推荐采用单体应用架构，因为其复杂性更低，风险也更低。随着团队规模越来越大，开发效率逐渐受限，而此时架构师对业务领域越来越清晰，此时可以考虑逐步的拆分应用。因为此时架构师已经更加熟知业务，而且团队也有人力资源来做这件事情。
